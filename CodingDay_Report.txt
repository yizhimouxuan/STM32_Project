Week4.5 Coding Day 项目报告

项目名称：
基于 STM32F446RE 与 RPLidar C1 的 LiDAR 数据处理与小车控制系统

一、项目概述

本项目基于现有 STM32F446RE 小车平台展开，核心目标是完成 RPLidar C1 激光雷达的数据接收、流式解析、距离提取，并将解析结果应用于小车的迷宫控制与调试输出。

根据《Week4.5-CodingDay.pptx》的要求，本次 Coding Day 的理想展示架构应当是基于 FreeRTOS 的软件管道，即通过中断驱动、任务间通信和任务调度，完成 LiDAR 数据从 DMA 接收、中断通知、任务级解析到任务级输出的完整流程。

但结合本组当前项目进度与系统状态，我们已经在现有工程上完成了 LiDAR 数据链路与小车控制逻辑的深度联调。当前实现方案已经稳定支撑距离滤波、姿态采集、迷宫控制和蓝牙调试输出，因此本组在实现层面保留了当前成熟方案，并在文档与答辩中完整分析了标准 FreeRTOS 架构的设计思想、优势和适用场景。


二、项目背景与任务目标

1. 项目背景

本组已有的嵌入式工程已经完成以下基础功能：

- 使用 USART3 与 RPLidar C1 通信
- 使用 DMA 接收 LiDAR 连续字节流
- 对接收到的扫描数据进行解析
- 提取前方、右侧、右前和左侧扇区距离
- 将 LiDAR 距离结果用于小车迷宫控制
- 通过蓝牙串口和 OLED 输出调试信息

因此，本项目并不是从零开始的雷达驱动实验，而是在现有稳定系统基础上，围绕 Coding Day 的要求，对系统架构进行分析、论证与整理。

2. Coding Day 目标

根据课程要求，理想的软件管道应当包括以下环节：

- UART + DMA 连续接收 LiDAR 数据
- DMA 半缓冲或满缓冲时触发中断
- ISR 只发送一个小型块描述符到队列 1
- LiDAR 解析任务从队列 1 取出描述符并解析 DMA 缓冲区中的原始数据
- 解析任务将结构化结果发送到队列 2
- 输出任务从队列 2 取出结果并打印或展示

该设计重点考察：

- 中断驱动事件处理
- 任务间通信
- 任务阻塞与唤醒
- 任务优先级设计
- 连续字节流的任务级解析


三、系统硬件与软件平台

1. 硬件平台

- 主控芯片：STM32F446RE
- 激光雷达：RPLidar C1
- 通信方式：USART3
- 数据接收：DMA
- 调试输出：USART1 蓝牙串口、OLED 显示屏
- 运动控制相关外设：编码器、电机驱动、MPU 姿态模块

2. 软件平台

- 开发语言：C
- 底层驱动：STM32 HAL
- 工程结构：CubeMX 生成基础工程 + BSP 功能模块
- 当前运行模式：裸机主循环 + 定时中断 + DMA + 回调处理


四、当前工程的实际实现

1. LiDAR 数据接收

当前工程使用 USART3 与 RPLidar C1 进行通信，波特率为 460800。接收端使用 DMA 缓冲区，缓冲区长度为 2048 字节。

对应代码位置：

- Core/Src/usart.c
- 关键变量：lidar_rx_buf
- 关键函数：LIDAR_UART_Start()

当前采用的是 HAL_UARTEx_ReceiveToIdle_DMA() 方式，即 DMA 持续接收数据，当 UART 出现空闲事件时，触发接收完成回调。

2. 接收回调与解析入口

当接收到一帧数据后，HAL_UARTEx_RxEventCallback() 会被触发。在该回调函数中，程序将实际收到的数据长度记录下来，然后直接调用 Lidar_ProcessFrame() 对本次接收到的数据进行处理，最后重新启动接收。

对应代码位置：

- Core/Src/usart.c
- 关键函数：HAL_UARTEx_RxEventCallback()

这说明当前工程的数据链路为：

USART3 + DMA + ReceiveToIdle -> 回调函数 -> Lidar_ProcessFrame() -> 距离提取 -> 控制层使用

3. LiDAR 数据流解析

LiDAR 解析逻辑位于 BSP/Src/lidar.c 中。当前工程已经支持连续字节流解析，而不仅仅是单包解析。

主要表现为：

- 使用 lidar_node_carry 保存跨块残留字节
- 使用 lidar_parse_work 作为解析工作区
- 使用 lidar_parse_scan_nodes() 解析标准 SCAN 节点
- 根据角度窗口提取前方、右侧、右前和左侧距离

这说明当前工程已经考虑了 LiDAR 数据流连续到达、节点跨块、原始流不按整齐边界对齐等实际问题。

4. 控制层使用 LiDAR 数据

LiDAR 解析完成后，系统不会直接把原始点云传给控制层，而是输出结构化距离结果。main.c 中通过以下函数继续使用这些数据：

- RefreshLidarDistances()
- MazeControlTask()
- Control()
- Send_Data_EverySecond()

其中：

- RefreshLidarDistances() 对原始距离进行滤波与有效性保持
- MazeControlTask() 根据前方、右侧、右前和左侧距离做迷宫决策
- Control() 负责切换手动控制与迷宫控制
- Send_Data_EverySecond() 负责通过蓝牙串口输出系统状态

这表明 LiDAR 模块已经不再是单独的传感器实验，而是已经直接进入小车行为控制层。


五、Coding Day 推荐架构分析

课程推荐展示的标准架构如下：

RPLidar C1
  ->
USART3 + DMA 循环接收
  ->
DMA Half/Full Interrupt
  ->
ISR 仅发送 block descriptor 到 Queue1
  ->
LiDAR Parser Task
  ->
解析结果发送到 Queue2
  ->
Output/Debug Task

该架构的核心思想是三层分离：

- 中断层只负责通知
- 解析层只负责处理原始数据
- 输出层只负责展示与调试

这种分层可以很好地展示实时系统设计原则，也能帮助说明为什么 ISR 应当尽量短、为什么解析任务优先级应高于输出任务。


六、两种传输方法的优劣分析

本项目当前稳定运行的方法为：

方法 A：USART3 + DMA + ReceiveToIdle + 回调中直接解析

课程更鼓励展示的方法为：

方法 B：DMA 半缓冲/满缓冲 + ISR 只发描述符 + Queue + Parser Task + Output Task

1. 方法 A 的优点

- 实现简单，联调路径短
- 与当前工程已经深度集成
- 当前 lidar.c 已经支持连续字节流解析与残留字节拼接
- 已经和 main.c 中的距离滤波、迷宫控制、蓝牙调试形成稳定链路
- 不需要引入额外的 RTOS 任务调度和队列管理
- 更适合当前项目处于整车联调阶段的实际情况

2. 方法 A 的不足

- 中断回调与解析逻辑之间的分层不够彻底
- 不利于完整展示 RTOS 中的 BLOCKED、READY、RUNNING 状态切换
- 不如标准 FreeRTOS 架构那样便于课堂展示

3. 方法 B 的优点

- 架构分层清晰
- 完全符合 Coding Day 对 ISR、解析任务、输出任务分离的要求
- 适合展示任务优先级、阻塞与唤醒、队列通信
- ISR 更短，更符合实时系统设计原则
- 解析与输出解耦，扩展性更好

4. 方法 B 的不足

- 需要对 usart.c、stm32f4xx_it.c、lidar.c、main.c 做较大范围重构
- 需要新增任务、队列、任务栈和更多系统对象
- 会改变当前 LiDAR 数据进入控制层的时序
- 有可能破坏已经联调稳定的小车控制功能


七、本组的工程取舍

经过评估，本组选择保留方法 A 作为当前主实现方案，同时在报告和答辩中完整论证方法 B 的设计思想与适用性。

做出这一取舍的主要原因如下：

1. 项目已经进入控制层联调阶段

当前系统已经不只是“接收到 LiDAR 数据”，而是已经把 LiDAR 的距离结果实际用于：

- 迷宫决策
- 电机控制
- 运动状态切换
- 蓝牙串口调试输出

一旦此时对底层传输架构做较大改动，就不仅仅是改接收部分，还会影响：

- 解析入口
- 数据时序
- 控制层取数方式
- 调试链路
- 小车实际运动表现

2. 当前方案已经能够支撑项目目标

当前方案虽然不完全符合标准 FreeRTOS 展示形式，但已经满足以下工程目标：

- 连续接收 LiDAR 数据
- 解析流式扫描节点
- 提取关键扇区距离
- 为控制逻辑提供实时输入
- 支持在线调试与状态观测

3. 当前阶段重构风险较高

对于整车系统而言，风险不在于“能不能把代码改出来”，而在于“是否会影响已稳定运行的功能”。在 Coding Day 时间限制下，继续重构底层架构可能带来较大联调成本，不利于整体系统稳定性。


八、关键代码模块分析

1. Core/Src/usart.c

该文件负责串口初始化、DMA 配置以及 LiDAR 数据接收。

需要关注的内容：

- USART3 的波特率配置为 460800
- DMA 接收缓冲区长度为 2048 字节
- LIDAR_UART_Start() 启动 ReceiveToIdle DMA 接收
- HAL_UARTEx_RxEventCallback() 作为当前解析入口

2. BSP/Src/lidar.c

该文件负责 LiDAR 数据处理，是整个 LiDAR 模块的核心。

需要关注的内容：

- Lidar_InitSequence() 负责启动扫描与初始化状态
- Lidar_ProcessFrame() 负责区分响应描述符与数据流
- lidar_parse_scan_nodes() 负责解析标准 SCAN 节点
- lidar_node_carry 负责跨块残留字节拼接
- Lidar_GetFrontDistanceM() 等函数向控制层提供结构化距离结果

3. Core/Src/main.c

该文件负责系统初始化、控制层逻辑和调试输出。

需要关注的内容：

- main() 负责系统初始化和 LiDAR 启动
- ReadSensor() 汇总编码器、姿态与 LiDAR 距离
- RefreshLidarDistances() 对距离进行滤波
- MazeControlTask() 实现迷宫逻辑
- Control() 负责控制模式切换
- Send_Data_EverySecond() 输出系统状态

4. Core/Src/stm32f4xx_it.c

该文件负责中断入口。

当前在 LiDAR 部分主要体现为：

- DMA1_Stream1_IRQHandler()
- USART3_IRQHandler()

在当前实现中，中断服务仍主要通过 HAL 回调链路进入接收与解析逻辑。


九、测试与运行效果

当前系统已经实现如下运行效果：

- LiDAR 可以稳定启动并持续输出扫描数据
- DMA 能够持续接收 LiDAR 字节流
- 系统可从连续流中提取有效扫描节点
- 程序可计算前方、右侧、右前和左侧距离
- 距离结果可用于迷宫控制
- 蓝牙串口可周期性输出调试信息
- OLED 可显示部分状态与测距结果

从工程运行状态来看，当前方案已经能够支撑小车控制与系统调试，具备较强的实用性。

十一、结论

本项目已经在现有 STM32F446RE 小车平台上完成了 LiDAR 数据接收、流式解析、距离提取与控制层联动，具备较完整的工程实现基础。

从课程要求出发，基于 FreeRTOS 的标准软件管道更加规范，也更适合展示中断驱动、任务调度和队列通信等实时系统概念。对此，本组已经在文档和答辩中给出完整分析，并能够说明该方案的优势、数据流设计和任务分层逻辑。

但从工程实现角度出发，当前系统已经进入小车逻辑控制层联调阶段，LiDAR 数据链路与控制层高度耦合。如果在此时做大幅重构，风险主要在于破坏现有稳定功能，而不只是代码改动本身。

因此，本组最终选择保留当前已经稳定运行的实现方案，并以文档分析和答辩说明的方式，补充对标准 FreeRTOS 架构的理解与论证。这一取舍体现了我们在课程要求与工程稳定性之间所做的综合平衡。
